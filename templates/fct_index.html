Four Coloring Simulator

Proof of the Four Color Theorem

The four color theorem states that any planar graph can be colored with no more than four colors in such a way that no two adjacent regions share the same color. This theorem was first proved in 1976 by Kenneth Appel and Wolfgang Haken using computer-assisted methods. Its proof was one of the first major results to rely on computer-assisted calculations, marking a milestone in the use of computational methods in mathematical proofs.

In high school I had written a paper for a class regarding the four color theorem, and in the process I stumbled upon an algorithm developed in the 1800s by mathematicican, Peter Guthrie Tait. It had inspired me to pursue a potential novel proof of the four color theorem without computer-assisted methods. For years, I consulted professors and peers seeking insight into the issue with my proof, as I was deeply skeptical that such a simple proof would have evaded the countless mathematicicans endeavoring to solve this famous problem. Upon recieving no input, I figured that a more hands on approach of implementing the algorithm would reveal the error.

When I first started and mapped out how my algorithm would work, my optimism had gotten the best of me. I read up on efficient triangulation methods that would improve my naive O(n2) approach by using a Constrained Delaunay Triangulation algorithm for an optimized O(n log n) time complexity. I learned that the current fastest algorithm for four coloring was O(n2), meaning that as long as the rest of my algorithm was within O(n log n), I would have developed a cutting edge algorithm. Based on my paper, I had considered this completely feasible. Such optimism quickly eroded as I ran into problems leading me to finally discover a glaringly obvious, in retrospect, error in my paper. For those curious, the error was overlooking an invalid case located in the induction proof.

This has not deterred my efforts however, as I continue to find a solution to remedy this error. In the mean time as bandaid solution, I implemented a backtracking algorithm which ostensibly emulates the core idea of the algorithm. The paper and instructions on how to use the tool is below. Using the Tool

  • Click anywhere within the constraints to make a node.
  • Drag between nodes to create edges.
  • Nodes must be a radius apart so that edges are visible.
  • Edges cannot intersect nodes or edges.
  • The Undo button will remove the last node or edge added.
  • The Clear button will remove all nodes and edges.
  • The Color button will color all nodes by applying the algorithm.
